# 行情、特征与运行调度

## 4.2 行情与数据

- 币安 REST：合约发现、交易规则、K 线、资金费率、盘口、24h ticker、持仓量。
- 币安 WebSocket：K 线、mark price、盘口、ticker、账户/订单用户数据流；
  自动重连、心跳、去重、乱序处理、断线后 REST 回补。
- 历史下载：K 线与资金费率分页拉取；K 线同时保留基础成交量与交易所原始计价成交量，资金费率按
  实际结算 K 线映射。
- 本地缓存：历史 K 线以精确区间存为 Parquet（可在数据管理中清除）；当前 v2 缓存 Schema 包含
  `quote_volume` 并使用独立版本路径，旧缓存不会在缺字段时被当成新数据读取。
- 正式引擎在每个实际决策批次自动保存完整 `MarketSnapshot`、账户起始状态与合约规则，作为含盘口
  回测的唯一数据源；不再运行独立盘口采集器。普通历史回测只读取 K 线，精确回放语义见
  [回测](backtesting.md)。
- 正式 `MarketSnapshot` 还从币安生产公开 Futures Data API 尽力读取最近两根已收盘 5m 的
  `open_interest_change_5m`、全市场账户多空比、大户持仓多空比和 taker 买卖量比及其变化。
  比率只描述账户/持仓拥挤度和主动成交，不代表“散户”“庄家”身份；OI 与比率任一端点失败时只
  省略对应实验字段，不能用 0 冒充，也不影响 K 线、盘口等核心快照。字段随正式快照冻结供影子
  分类与精确回放，但在形成可复核样本前从标准外部 Provider payload 中剔除，`trend-v2` 同样不
  用它们改变订单。币安只提供最近 30 天的这些统计，因此系统不声称普通任意历史窗口可以重建。

### 独立 AI 行情分析数据包

- 这是独立研究模块；默认按需分析不复用正式引擎的 `5m/15m/30m/1h/4h + 日线结构` 特征阶梯，也不
  改变正式决策、风控或下单。用户可显式选择“AI 分析辅助 · SHADOW”，让正式周期改用同一套
  Kansoku 风格数据包和分析 Schema 生成意图，但该模式只记录硬风控结果且永不调用 Broker。
  数据包只取 `5m/15m/1h`：每周期最多 150 根已收盘 K 线用于
  指标预热，冻结最近 60 根原始 OHLC、基础成交量与计价成交量；15m 额外取 500 根计算按 UTC
  自然日、同一时点对齐的最近 5 日相对成交量。5m 负责时机，15m 负责结构与主锚点，1h 负责趋势。
- 三周期独立计算 EMA(9/21/55)、MACD(12,26,9)、半径 3 的已确认摆动点；5m/15m 计算 UTC
  日内 VWAP。数据包还冻结 mark/index/bid/ask/24h 成交额、资金费率、基差、未平仓量、20 档盘口
  不平衡、最近 1000 笔主动成交不平衡、BTCUSDT/ETHUSDT 对照、测试网账户/本标的持仓以及上一份
  成功分析。BTC/ETH 是加密市场对 Kansoku SPY/QQQ 背景的显式适配，不冒充美股数据。
- 当前没有新闻、事件日历和 Kansoku 非公开 Pro 检测器；数据包用 `unavailable_inputs` 与
  `pro_detectors_available=false` 明示缺口。期权结构来自 Deribit 无认证公共 API：程序每小时读取一次
  全部活动期权元数据、每 5 分钟按实际结算币批量刷新市场汇总，并用共享锁让同批候选复用一份快照，
  不按候选或行权价重复请求。外部请求只读取全市场公共数据，不携带账户、持仓或 Provider 输入。
- 期权覆盖不是 BTC/ETH 固定白名单。程序从活动合约的 `base_currency` 动态发现标的，只有同时满足至少
  两个到期日、六个有持仓合约、Call/Put 均有持仓和近现价覆盖、24h 成交量非零、至少四个 IV 与有效
  报价的确定性质量门槛，才输出该标的自身的 OI、Put/Call OI 比、24h 量、前三个近到期结构、ATM IV
  期限变化和 Call/Put 最大 OI 行权价；否则只披露不合格原因。每个候选同时携带质量合格的 BTC/ETH
  期权作为 `market_benchmark`，并用 `direct` 明确标记自身期权是否可用，禁止把基准冒充本币数据。
  OI 单位保持 Deribit 返回的标的基础币数量；它不是带符号的交易者或做市商仓位，大 OI 行权价不是
  已确认支撑阻力，Put/Call 和 IV 也不是独立方向信号。数据源超时、拒绝或返回无效数据时，数据包降级
  标记期权暂不可用，既不猜值也不使单标的或整批分析失败。
- 模型必须把所有缺失输入视为未知，不能把空值写成“没有新闻/事件/期权风险”。分析保存的是本次冻结
  输入，包括期权快照时间、质量统计、角色与解释边界；详情页/API 不会事后重拉数据改写当时图景。
- 分析结果保留英文 JSON 键、方向枚举、币种、周期、数字和 EMA/MACD/VWAP/T1/T2 等标准缩写，
  但摘要、锚点理由、情景、计划、证据和缺失数据影响等面向用户的自然语言由模型直接输出简体中文。
  已保存的旧结果保持原文，不在读取或展示时调用翻译模型改写审计记录。
- 批量研究直接读取正式引擎同一候选池的前 `candidates_per_cycle` 个标的，保持排名与去重顺序；这里
  共享的是“候选标的”而不是正式执行集合，因此不会追加当前持仓。候选池为空时与正式引擎一样先
  刷新全市场，仍无合格标的则拒绝创建。每个标的独立冻结数据包、调用 Provider 并保存分析记录，
  队列按候选顺序串行推进；单项失败不阻断后续项，取消任一活动项会取消整个当前批次及尚未开始项。
- 自动研究由用户在前端显式启用，后端从下一个 UTC 15 分钟边界开始，每 15 分钟读取同一批正式候选；
  每个标的仍使用独立 Prompt 串行发送，不改成一个多标的大请求。每轮先读取同标的最近成功分析：
  neutral 观望可在下一轮重新评估；方向计划则先按需刷新结果，只有观望、未入场先到 stop/T1、T2、
  结构止损、T1 后余仓保本或同 K 线顺序不确定等终态才允许创建新分析。顺序不确定仍保留原结果，
  但不会继续阻塞后续研究。等待入场、active、T1 部分止盈或结果刷新失败均跳过，避免同一标的叠加
  相互冲突的活动计划。上一轮跨过新边界、正式
  引擎或其他模型任务占用时跳过该轮，不并发积压；下一边界继续尝试。停止自动研究只取消未来轮次，
  已创建的一轮继续完成或由现有单项/批次取消入口终止。该开关只存在于当前后端进程，服务重启后
  默认关闭，防止未经再次确认自动消耗外部模型额度。
- 计划结果按需刷新，观察窗口从分析完成后的**下一根完整 5m K 线**开始。方向性计划只有先触及 entry
  才进入 active；等待期间若先触及结构 stop 或 T1，分别以 `stopped_before_entry`（未入场止损）或
  `target1_before_entry`（未入场 T1）结束观察，不冒充成交、亏损或止盈。入场后依次识别结构止损、
  T1 部分止盈、T2；
  T1 后按约减半并把余仓管理参考移到 entry，可形成“T1 后余仓保本”或“T2”。同一根 5m K 线同时
  触及会导致不同结论的多个价位时，结果评估只补取该 5m 窗口内时间严格连续的五根完整 1m K 线，
  按分钟顺序重放同一状态机；这只是事后结果细分，不进入 AI 的冻结数据包或 Prompt。若冲突价位落在
  不同分钟即可确定路径，并在结果说明中披露使用了 1m；若五根数据不完整，或冲突仍发生在同一根 1m
  K 线内，OHLC 依然无法证明分钟内顺序，结果必须保持 `ambiguous`，不能用有利或不利顺序猜测。
  `bars_observed` 始终保持完整 5m K 线根数口径。neutral 只记为观望观察，不虚构交易结果；程序
  确定性生成的结果说明同样使用简体中文。
- 结果既可按单条刷新，也可把前端当前加载的最多 30 条已完成历史作为一批串行刷新；批量请求去重
  记录 ID，单条不存在、未完成或行情读取失败只进入该批错误清单，不阻断其他记录。每条结果仍独立
  保存，批量动作不调用 Provider、Broker 或正式引擎。
- 历史表现只纳入已经入场且结果可确定性定价的 `stopped`、`target2` 与
  `breakeven_after_target1`。固定名义金额口径用“金额 / entry”作为每笔数量；固定止损金额口径用
  “止损金额 / |entry-stop|”反推每笔数量。T2 按一半仓位在 T1、一半在 T2 结算；T1 后保本按一半
  在 T1、一半在 entry 结算；结构止损按全仓 stop 结算。胜率统一按正盈亏笔数 / 已结算笔数，零盈亏
  计入分母但不计胜。未入场 stop/T1、neutral、等待入场、active、T1 部分止盈和 `ambiguous` 均不
  冒充已结算交易；费用、滑点和资金费没有实际成交依据，因此明确不纳入该研究测算。
- 辅助模式仍按正式引擎候选与已有持仓构成同一批次，用一次物理 Provider 调用返回等长同序分析。
  新方向只有置信度不低于 0.55 才转换为开仓；同向持仓不自动加仓，反向分析只生成平仓而不在同轮
  反手。程序固定 1 倍杠杆和 1% 风险申请，T1 写入正式止盈字段，T2 仅随分析记录进入影子结果评估。
  情景概率要求使用 0–100 百分点；模型若整组使用 0–1 且总和约为 1，程序确定性换算为百分数后保存，
  其他错误概率继续使整批失败，不影响价格、方向或执行提示的严格校验。
  neutral 的观望区间必须围绕锚点；仅有边界轻微遗漏且锚点偏离不超过原区间宽度时，程序把边界扩展
  至锚点以免纯展示字段拖垮整批，明显脱离当前结构的区间仍严格拒绝。
  方向性分析缺少开仓所需的 order/setup/trigger/invalidation/target，或 `LIMIT` 缺少 TTL 时，仅该标的
  降级为 `HOLD`；程序不合成交易参数，其他结构合法的同批标的继续各自进入转换与硬风控。
  T1 盈亏比不足不是批量格式错误：程序保留模型原价，正式辅助模式由下游硬风控按实际刷新与精度对齐
  后的 `>1.15R` 门槛逐标的否决，独立研究则保留该弱计划供复盘。

## 4.3 选币与特征

- 动态全市场扫描 USDT 永续，按上市时间、完整度、价差、成交额、波动率、趋势过滤排名；
  启动即扫，之后每分钟自动刷新轮换；每周期默认向所选决策 Provider 提交排名前 5 个候选，
  可配置为 1–20 个。
- 特征：**5m/15m/30m/1h/4h** 共享 EMA、RSI、ATR、收益率、成交量，以及 20/50 根区间高低点与
  区间内位置（`range_position_50`，0 为区间底、1 为区间顶）。每个周期另给出排除最新已收盘
  K 线的前 20 根边界、最新收盘是否真正突破/跌破该旧边界、两侧各两根已收盘 K 线确认且不会重绘
  的最近摆动高低点及距今根数、最新 K 线收盘在自身高低区间的位置；另有基差、持仓量、
  20 档盘口与近期成交失衡等微观结构特征。每个周期获取最近 200 根 K 线、排除未收盘 K 线
  后计算，连同日线结构位合计 141 个快照特征；LLM 接收特征值而不是原始 K 线数组。
  5m 负责入场时机，15m/30m 负责可执行走势确认，1h/4h 定义更高周期趋势与结构背景；分析周期
  只决定何时重新审视这套完整阶梯，不允许模型因本次是某个周期而忽略其他层级。
- **日线只给结构位**（`1d_range_high_20` / `1d_range_low_20` / `1d_range_position_20`，以及
  `1d_previous_high` / `1d_previous_low` / `1d_previous_close`），
  **不套用完整特征阶梯**。理由有三：(1) 日线的价值就是那两个位——20 日高低点是真有挂单
  堆积的地方，而 50 分钟高点没人防守；日线 RSI/成交量比又会变成没有规则读的读数。
  (2) K 线根数才诚实：`MarketScanner.minimum_listing_days = 30` 只保证 ~29 根已收盘日线，
  若照搬 `ema_50` 会在上市不足 50 天的币上**静默退化成普通均值**，`1d_ema_spread` 就成了
  EMA20 对 SMA29 的错误比较。(3) 区间位以**实时 mark 价**而非昨日收盘计算（收盘可能已陈旧
  近 24 小时），且**刻意不做钳制**——大于 1 表示价格已突破 20 日高点，那是信号不是错误。
  Prompt 明确日线是**位置而非趋势**，不得读成方向性偏好；与日内位冲突时以日线位为准。
- **准入耦合**：日线结构位要求至少 `DAILY_STRUCTURE_PERIOD`（20）根已收盘日线，这正由扫描器
  30 天上市门槛保证；两者的关系有测试钉住（门槛降到 20 天以下会让新币在
  `market_snapshot` 阶段抛错并整体掉出该周期）。
- **喂给决策 Provider 的周期 = 规则用到的周期**（`DECISION_FEATURE_INTERVALS`）。1m **不进入**决策
  payload：入场形态明确使用 5m/15m/30m，趋势与结构背景明确使用 1h/4h，没有一条规则读 1m，
  而没有分配角色的数据比
  没有数据更糟——模型必然自行发明用法，且每次调用可能发明得不一样。1m 还会双向干扰规则
  （`1m_ema_spread` 一小时翻多次符号，既可能误伤有效的高周期入场，也可能给勉强的入场背书），
  且 LLM 不会按周期重新校准 RSI 70/30 这类阈值。"现在这一刻"由微观结构与 mark/bid/ask 回答，
  "刚才是否刚拉过"由 `5m_return_1` 回答。测试网持仓直接从交易所账户快照盯市，不存在使用
  1m 空特征快照的模拟持仓路径。新增决策周期必须同时补上读它的规则。
- 区间高低点是模型判断"是否收复/拒绝参考位"的依据——均线只说方向、不说位置。Prompt 逐条点名
  哪个字段回答哪个入场条件，并要求：若某形态所需证据不在 payload 中，该形态即视为不成立。
  `range_high_20` 包含最新 K 线，只适合描述当前区间；`prior_range_*` 与
  `breakout_above_20` / `breakdown_below_20` 只说明最新一根收盘已越过此前边界，供诊断使用。
  可交易的趋势突破必须读取 `breakout_hold_above_20` / `breakdown_hold_below_20`：最后两根已收盘
  K 线都越过同一个、在它们之前计算的 `breakout_hold_high_20` / `breakout_hold_low_20`，避免把
  瞬时刺穿当成确认，也避免第一根突破 K 线移动第二根用来验证的边界。`last_swing_*` 只有在对应
  `*_confirmed=1` 时才是已确认摆动点，避免让模型把仍会重绘的末端极值冒充结构。依赖未来突破的
  决策必须 `HOLD` 到两根确认 K 线都收盘；依赖未来收复的市价单同样必须 `HOLD`，其他明确限价
  触发点才可进入本地待触发流程。
- **「延伸」只由 `ema20_distance_atr` 定义**（价格离本周期 20 根均线多少个 ATR，带符号；
  约 ±2.5 以上视为追高）。**区间位不是延伸**——这是一个曾经让系统 170 次推理 0 次下单的
  逻辑陷阱：`range_position` 量的是价格在滚动区间里的位置，而**任何持续趋势按定义就会把价格
  推到自己区间的边缘**（上涨创新高→位置≈1）。把边缘读成"延伸"，等于规定"只在没有趋势时做
  趋势入场"，前提与条件直接冲突。实测：同向趋势样本中，两道区间边缘否决误伤 76/119，
  而真正离均线 ≥2.5 ATR 的只有 7/119（中位数仅 +0.26 ATR）。
- **日线位是参考位，不是否决闸**：带同向趋势穿过日线极值 = 突破，**利好该方向**而非禁止；
  逆势逼近前方日线位 = 容易受阻（形态 (3) 的拒绝入场点），也可作为形态 (1) 的收窄目标依据。
  形态 (2) 视日线位守住为比日内位更强的证据。
- 每周期特征只带一次前缀（`5m_rsi_14`），不再同时给出无前缀副本，避免同一读数被当成两份独立证据。
- `recent_trade_imbalance` 取自定量条数的成交流水，覆盖时长随标的活跃度变化，
  因此同时给出 `recent_trade_seconds`；Prompt 要求把过短的窗口当噪声而非订单流。
- EMA 以前 `period` 个值的均值做种子并跑满全部已获取历史，避免窗口首值的噪声残留在读数里。
- 唯一选定的 5m/15m/30m/1h/4h 分析周期按对应 UTC K 线边界对齐调度；每周期把候选池前 N 个
  标的与全部已有持仓去重后组成一次批量分析，确保掉出候选池的仓位仍会获得主动
  `HOLD/ADD/REDUCE/CLOSE` 决策；同向增仓须显式 `ADD`。
- 每个周期先串行采集各标的快照，再进行一次批量 Provider 调用，最后按输入顺序逐标的风控与执行；整轮另受该周期自身的绝对时长预算约束；到下一周期边界仍未
  完成时取消剩余工作，不允许一个慢请求永久吞掉之后所有周期。`/api/status.scheduler` 实时返回
  当前周期、标的、阶段（行情快照/账户/决策）、完成数、开始时间及最近调度/选币/看护错误，
  前端运行区将阶段明确显示为“准备合约规则 / 采集行情 / 读取账户状态 / N 个标的批量分析中”；
  批量调用期间不再把空的单标的字段显示成“准备中”，也不展示整批返回前恒为 `0/N` 的伪进度。
- **分析周期单选**：用户只能从 5m/15m/30m/1h/4h 中选择一个（默认 15m）；1m 既不触发
  LLM 决策、也不进入 LLM 特征（见上），并且不属于正式运行路径；
  默认取自 `CANDLEPILOT_CADENCES`，也可在前端运行前经 `POST /api/cadences` 修改，
  运行时锁定。后端配置、API 与引擎都会拒绝零个或多个分析周期，只有被选中的周期会启动调度任务。
- **每周期候选标的数**：每个周期选取候选池排名前 N 的标的，N 可配置（默认 5，范围 1–20）；
  已有持仓不占用这 N 个候选名额，而是在去重后额外加入实际分析集合。
  默认取自 `CANDLEPILOT_CANDIDATES_PER_CYCLE`，也可在前端运行前经
  `POST /api/candidates-per-cycle` 修改，运行时锁定。

## 4.3.1 运行看护与自动停止

调度器有一个独立于决策周期的看护任务（默认每 5 秒轮询），命中任一条件即**优雅停止**引擎
（停止产生新决策、保留现有持仓；测试网持仓仍由交易所侧止盈止损括号单保护），并把原因写入
`/api/status` 的 `auto_stop_reason`，前端顶部横幅展示。独立轮询是必要的：否则 4h 周期会把
一次到期停止拖延数小时。停止原因不会阻止重新启动，`start` 时自动清空。
同一轮看护还读取测试网持仓、标记价和交易规则，维护移动止损并运行固定的部分止盈影子实验；
后者只持久化 1R 部分成交与余仓保本的可执行模拟结果，不修改订单，也不参与自动停止或亏损熔断。
优雅停止先取消并等待调度任务，再结束运行会话；引擎在每个批次结果、待触发限价意图和最终订单提交前
都会重新检查运行状态、自动停止原因与紧急锁，停止请求生效后不允许同批后续标的继续提交订单。
自动紧急停止则必须先取消并等待全部周期决策退出，再撤单和平仓；禁止在仍有 Provider 调用或
预交易任务存活时先平仓，否则迟到的决策可能在账户清空后重新开仓。

- **运行上限**（启动前设定，运行时锁定；`POST /api/run-limits`，默认取自
  `CANDLEPILOT_MAX_RUN_SECONDS` / `CANDLEPILOT_MAX_RUN_COST_USD`）：
  - **时长**：本次运行超过 N 秒即停。
  - **预算**：本次运行的**等效成本**（复用 `/api/metrics/run-session` 的
    `equivalent_cost_usd`）达到 N 美元即停。订阅制（Codex/Claude）实际不按次计费，此处为按
    API 标准价的折算估算，仅作护栏；成本未知时**绝不**触发停止。
  - 两者均可留空表示不限；先到者触发。
- **Provider 连续失败**：不再按“不可用多少秒”停止，因为 1h/4h 周期在下一次调用前没有机会
  证明已经恢复。所选 Provider 调用失败计 1 轮，同一决策内按 5 秒、15 秒退避重试同一个 Provider；
  连续 3 轮仍失败时停止，成功即清零。Provider 抛出的未预期运行异常与协议错误、
  网络错误和超时一样进入失败审计及次数阈值，不能只变成永久重复的周期错误。看护任务读取次数阈值并执行优雅停止；状态
  API 同时返回 `route_failure_count` 与 `route_failure_limit`，不再返回时间戳。
- **紧急回补累计上限**：每次保护括号创建失败但 reduce-only 回补成功计 1 次；同一个
  `live_run_id` 累计达到 3 次时，在第 3 次执行审计完整落库后立即自动优雅停止，不平掉其他已有
  持仓。新运行从 0 重新计数。状态 API 返回 `rescue_count` 与 `rescue_limit`；运行中首次回补后
  前端顶部显示累计预警，停止后显示明确原因，历史运行分组也持久化同一停止原因。
