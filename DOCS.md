## 1. 如何阅读

所有任务都必须先完整阅读根目录 `AGENTS.md` 和本文件，再按“文档路由”完整阅读与任务相关的专题。
跨领域变更必须读取所有受影响专题；不确定归属时宁可多读相邻专题。无需为一次前端布局修改加载整份
VPS、回测和 Broker 细节，也不得只看本入口就修改某个领域的实现。

文档描述与代码不一致属于缺陷：先以仓库当前实现、测试与数据库迁移的实际状态核实行为，再在同一
变更中修正代码或文档，不能为了省事保留两套互相矛盾的“权威”说法。

## 2. 概述

CandlePilot 是一个**本地优先、可审计**的日内交易系统，当前专注于币安
USDⓈ-M USDT 永续合约。外部 LLM 或本地确定性策略生成结构化 `TradeIntent`，确定性风控拥有
最终否决权。

- 唯一交易环境是**币安官方 Demo 测试网**；回测是按需分析，不是运行模式。
- **没有真钱实盘入口。**真钱实盘、美股/券商、新闻、社交和链上数据均不实施。
- 工程验收关注无前视偏差、可复现、可审计、风险边界和故障恢复，不以盈利为验收条件。

## 3. 不可破坏的系统边界

- 必须配置 Demo 测试网 Key/Secret；缺少凭据或 `.env` 残留已移除的 `CANDLEPILOT_MODE` 时拒绝启动。
- Broker 硬性拒绝生产交易地址。测试网凭据不进入 LLM 子进程、数据库或日志。
- 后端只允许监听 localhost。VPS 由同机 Nginx 通过 HTTPS 暴露，并强制启用控制台鉴权与 Secure Cookie。
- 外部 CLI Provider 在隔离临时目录内运行，仅接收白名单环境变量；禁用工具、文件、Shell 与模型侧网络工具。
  本地规则不启动子进程、不访问网络。
- 已通过控制台认证的用户可在模型接入卡片启动或取消 Codex CLI 设备码登录并二次确认登出；后端只
  返回 OpenAI/ChatGPT HTTPS 授权地址、一次性代码和阶段状态，不返回 CLI 原始输出或认证令牌。
- Codex CLI 登录有效时可通过受控 app-server 调用读取额度；后端不读取认证文件，只返回套餐、
  实际额度窗口、剩余百分比与重置时间。前端不按套餐臆测不存在的周/月窗口。
- 开仓和加仓必须经过下单前行情/账户刷新、确定性风控及交易所规则校验，并同时带交易所侧止损和止盈。
- `ADD` 必须请求与现有标的仓位完全相同的杠杆；风控在定量前拒绝变杠杆加仓，避免 Broker 重配整个
  标的后使旧仓位保证金越过单标的或组合上限。
- 独立 AI 行情分析可作为正式决策的可选 `shadow` 输入，但该模式永不提交订单：程序固定 1 倍杠杆，
  把模型计划 T1 映射为正式 `TradeIntent.take_profit`，仅保留 T2 做分析结果影子评估，并完整运行硬风控。
- `local-structure-shadow`、`local-flow-shadow` 与 `local-structure-flow-shadow` 是本地确定性实验
  Provider：正式运行即使通过全部硬风控也只审计、不调用 Broker；普通或正式回放回测仍模拟成交，
  用于与未改动的 `local-rule` 基线做消融比较。
- 任何带 `shadow_only=true` 的本地待触发限价意图在后续行情复核中都必须保留该执行边界；触价后
  只完成风控影子审计，不得因重新计算 `RiskDecision` 而升级为可提交 Broker 的订单。
- 正式单标的、批量与待触发限价统一经过同一最终执行资格门：有订单、风控接受、非等待且非影子
  四项必须同时满足，任何路径不得自行放宽。
- 纯减仓和平仓不能被盈亏比、亏损熔断或行情陈旧等新增风险护栏阻止。
- 风控放行不等于成交成功。订单、保护单、失败回补、用户流和紧急清理必须分别审计并在状态不明时
  fail closed。
- 紧急锁写入持久化状态，重启不清除；只有引擎停止且账户无持仓、无普通挂单、无 Algo 挂单时才能解除。
- `.env`、数据库、行情、模型登录态和 TLS 等运行数据不得被普通代码更新或 Git 更新覆盖。
- 网页软件更新先以独立非安装检查确认 GitHub `origin` 当前分支存在快进提交，随后才允许用户另行
  确认安装；检查不改工作树、不启动 root 更新器，安装仍执行备份、健康检查和失败回滚。
- 设置页可读取安装器生成的脱敏备份清单并删除过时备份；删除仍经受限 root 维护服务执行，只接受
  严格格式的清单内 ID，最新备份和仅存的一份备份始终受保护，应用进程不能选择任意文件路径。
- VPS 安装器把 CandlePilot 输出放入独立 systemd journal 命名空间；设置页只能通过受限 root
  维护动作清理该命名空间，不得删除 SSH、Nginx 或其他服务日志，也不能传入日志路径或命令。
- VPS 卸载器要求明确确认，默认删除经校验的实际 CandlePilot 备份根目录；路径从显式覆盖或 root
  持有的网页更新配置解析，宽泛系统目录与符号链接一律拒绝，共享系统包和防火墙规则保持不变。

## 4. 架构与关键数据流

- 后端：Python 3.12、FastAPI、Pydantic、SQLAlchemy、SQLite WAL；单进程异步，不依赖 Redis/Celery。
- 前端：React、TypeScript、Vite；REST + WebSocket，白色浅色主题。
- 存储：SQLite 保存业务、运行和审计数据；Parquet 保存大容量历史行情。
- SQLite ORM 表与迁移定义集中在 `storage.models`，数据库生命周期和通用审计仓库保留在
  `storage.database`，独立行情分析持久化由专用仓库模块承载；原有公开导入路径保持兼容。
- 核心边界：`DecisionProvider` 负责模型接入，`FeaturePipeline` 负责特征，`TradingEngine` 负责编排，
  `AggressiveRiskPolicy` 负责确定性风控，Binance Broker 负责唯一的测试网执行。

正式运行的关键路径：

1. 动态扫描币安 USDT 永续并与 Demo 测试网可交易合约取交集。
2. 在唯一所选周期收集候选与已有持仓的完整多周期行情，并读取一份共同账户快照。
   正式快照还尽力冻结币安生产公开市场最近两根已收盘 5m 的 OI、全市场账户多空比、
   大户持仓多空比与 taker 买卖量变化；任一可选统计失败只移除对应实验字段，不拖垮核心行情。
   这些字段当前只供确定性影子实验和正式回放，标准外部 Provider payload 不读取，避免未经验证
   就改变现有决策。
3. 本次唯一 Provider 用一次批量调用返回等长同序意图；三个本地结构/资金流实验 Provider 会在
   决策结果上声明强制 live shadow，只有标准 `local-rule` 继续具备测试网下单资格。可选 AI 分析辅助 SHADOW 改用同批
   `5m/15m/1h` Kansoku 风格数据包先产出完整分析，再确定性转换为 1 倍 `TradeIntent`；模型若按
   `0–1` 返回一组总和约为 1 的情景概率，程序只把该非交易关键字段确定性换算为百分数，其他错误总和仍拒绝。
   neutral 观望区间若只在锚点边缘外且偏离不超过原区间宽度，程序可扩展边界纳入锚点；明显脱离锚点仍拒绝。
   方向分析缺少任一开仓执行提示时只让该标的 `HOLD`，不猜测交易参数，也不使同批其他标的失败。
   T1 只要方向几何有效即可进入转换，低于目标盈亏比不会作为批量协议错误；正式硬风控仍逐标的要求
   下单前盈亏比严格大于 1.15:1。
   失败按固定退避重试同一 Provider。前端以
   互斥选中的卡片作为唯一来源，不再重复显示路由或“实际承载”摘要。
4. 每个非 `HOLD` 意图在下单前重新读取最新行情、账户和合约规则。
5. 确定性风控定量并决定是否放行；Broker 执行入场和保护流程。
6. 输入、Prompt、模型输出、风控、执行、用户流、移动止损和运行边界分别持久化审计；每次正式
   运行同时固化实际执行代码的 7 位 Git 提交号，更新后仍可追溯历史运行版本。
7. 正式运行收益按价格已实现、未实现与可归属手续费拆分；资金费无法可靠归属时明确标为未知。

回测复用正式运行的特征、Prompt 与风控代码，只模拟交易所负责的撮合、费用和资金费；资金费在对应
5m 结算边界立即进入现金、权益、可用余额和滚动 24 小时盈亏，平仓时只做交易归因而不重复扣款。所有决策只能
看到当时已经收盘的数据。普通历史 K 线同时保留交易所返回的基础成交量与精确计价成交量，不能用
`基础量 × 收盘价` 近似成交额特征；普通历史回测明确缺少盘口与订单流。完整微观结构输入只来自正式运行自动
保存的实际批量输入、起始组合和合约规则，并按原批次精确回放，不再提供独立盘口采集器。

独立 AI 行情分析与正式决策链隔离：它只在正式引擎停机且没有其他模型任务时，按需冻结
`5m/15m/1h` 原始已收盘 K 线及 Kansoku 风格技术摘要、相对成交量、衍生品资金流、BTC/ETH
市场对照、经质量门槛筛选的 Deribit 公共期权结构和当前账户上下文，再由唯一所选外部 Provider 返回方向/观望、锚点、2–4 个情景、结构
止损及显式 T1/T2；发给严格结构化输出 Provider 的 Schema 要求每个声明字段都显式返回，包括
允许为空的缺失数据影响列表。按需研究入口的分析结果只保存和展示，不转成 `TradeIntent` 或订单；用户显式选择的正式
AI 分析辅助 SHADOW 则走上文受限转换并完整执行硬风控，但仍不提交订单。期权标的由公开合约动态发现：质量合格时使用
标的自身期权，其他标的只把 BTC/ETH 期权作为大盘基准；持仓量不解释为有方向的交易者或做市商仓位，
大持仓行权价也不冒充已确认支撑阻力。新闻、事件、不可用的直接期权与非公开 Pro 检测器等缺失输入
必须作为未知项披露，不能解释为风险不存在；期权源故障只降级该上下文，不拖垮整批分析。前端“AI 分析”工作台
既可输入单个标的，也可把正式引擎当前候选池前 N 名作为一个队列批量分析；批量不追加已有持仓，
每个标的仍保存独立冻结输入、模型调用与结果。用户还可显式启用对齐 UTC 15 分钟边界的后端自动轮询；每轮仍按正式候选池逐标的独立发送，先刷新同标的最近方向计划结果，尚未形成终态时跳过；同 K 线顺序不确定也视为终态，允许下一轮重新分析。自动轮询服务重启后默认关闭，不会未经确认继续消耗额度。结构化 JSON 的键名、枚举和市场标准缩写保持英文，所有面向用户的分析文字由模型直接输出简体中文；历史记录不回写翻译。工作台提供任务取消/轮询、历史列表及方向筛选、三周期冻结 K 线及计划价位叠加展示；历史默认显示偏多、偏空和未完成任务并过滤观望，也可单独查看偏多、偏空或全部。左侧每条历史同时显示方向与最新计划结果，并可对当前加载的已完成记录批量更新结果。观望区间使用独立布局并与后续情景卡片保持正常间距；计划结果只读取分析
完成后的完整 5m K 线；入场前先触及结构 stop 或 T1 会分别固化为“未入场止损”或“未入场 T1”，
只有先触及 entry 才进入实际计划路径并继续判断 stop/T1/T2。只有 5m 结果出现顺序歧义时才补取对应窗口的
五根完整 1m K 线细分，1m 数据不完整或冲突仍在同一分钟内时继续标成不确定。工作台还按全部历史中
已入场且可明确结算的计划展示两套可调假设：每笔固定名义金额，以及按 entry 到结构止损距离反推仓位
的每笔固定止损金额；两者分别给出累计盈亏和胜率，且明确排除未入场、观望、进行中与顺序不确定结果。

## 5. 关键默认值

以下只是跨领域快速索引，完整语义与例外以对应专题为准：

| 项目 | 当前值 |
|---|---|
| 决策周期 | 5m / 15m / 30m / 1h / 4h 中唯一一个，默认 15m |
| 每周期候选 | 默认 5，范围 1–20；已有持仓额外加入 |
| 外部 Provider 默认超时 | 45 秒；正式运行可按次确认 1–600 秒 |
| 分析快照最大年龄 | 默认 75 秒 |
| 最大杠杆 | 10x |
| 单笔及单标的计划止损风险 | 不超过权益 1% |
| 组合计划止损风险 | 不超过权益 4% |
| 总 / 单标的初始保证金占用 | 不超过权益 80% / 10% |
| 下单前原始盈亏比 | 必须大于 1.15:1 |
| 滚动 24 小时亏损熔断 | 默认 5%，可设置 0.1%–50% |
| 结构门槛 | 默认 `shadow`；只有 `enforce` 会因此拒单 |
| 移动止损 | 默认 `shadow` 并行记录五组候选及首次模拟成交；`live` 只执行 2R/1R |
| 部分止盈实验 | 固定 `shadow` 并行比较 1R 止盈 25%/50%，剩余仓位按入场价模拟保本 |
| 净亏保护退出后同标的冷却 | 90 分钟；覆盖止损及手续费后亏损的止盈 |
| 止盈后重入实验 | 只观察 15/30/60 分钟等待窗口，不否决或修改订单 |
| 控制台会话 | 默认 7 天，范围 300–604800 秒 |

## 7. 配置、CLI 与验证入口

配置从当前目录 `.env` 加载，shell 中显式 `export` 的同名变量优先。完整变量表和 API 清单见
[配置、CLI 与 API](docs/configuration-and-api.md)。CLI 只保留：

- `candlepilot doctor`：检查 LLM 认证和币安公共接口，不下单。
- `candlepilot serve`：启动本地 API 与已构建前端。

安装、VPS 运维和验证细节见 [运行、验证与 VPS 运维](docs/operations.md)。仓库级 Python 命令必须从
根目录执行，前端命令必须从 `frontend/` 执行：

```bash
.venv/bin/ruff check .
.venv/bin/pytest -q
pnpm --dir frontend run test
pnpm --dir frontend run build
.venv/bin/python scripts/check_commit_messages.py --commit HEAD
```

## 9. 文档维护约定

1. 本文件是权威入口；本文件索引的专题共同构成权威文档集。未被索引的临时说明不具备同等地位。
2. 每个独立功能必须同时更新代码、测试、本入口中受影响的核心摘要，以及所有受影响专题。
3. 核心边界、跨领域默认值和文档路由留在本文件；实现细节只写入一个最合适的专题，其他位置使用链接，
   避免 README、入口和多个专题复制同一长段说明。
4. 只有测试与必要实际检查通过后，才能把功能描述为已实现。前端改动必须运行测试、构建和浏览器验证。
5. 提交格式、共同作者、提交后校验与立即推送规则以根目录 `AGENTS.md` 为准。
